# DDD领域驱动设计笔记

## DDD整体目标

围绕以需求，团队协作为中心，从业务建模到软件架构落地的一套思考体系

## 注意：DDD 是迭代过程

实施过程不是一次性线性流程。领域模型会随着对业务理解的加深不断修正：建模 → 实现 → 发现问题 → 重新建模，这个循环贯穿整个开发周期。

## 实施过程

### 0. 领域发现：Event Storming（事件风暴）

在划分领域之前，先通过 **Event Storming** 发现领域边界。以电商下单流程为例：

**Step 1：发散——贴出所有领域事件（橙色便利贴）**

全员参与，不加评判，把系统里所有"发生过的事"写出来，用过去时命名：

```
订单已创建 / 库存已锁定 / 支付已完成 / 物流已发货 / 订单已取消 / 优惠券已核销 ...
```

目标是穷举，不遗漏，不讨论对错。

**Step 2：排时间线**

把事件按业务发生的时间顺序从左到右排列：

```
优惠券已核销 → 订单已创建 → 库存已锁定 → 支付已完成 → 物流已发货
```

这一步会暴露流程中的争议点和盲区，团队会开始讨论"这两个事件谁先谁后"。

**Step 3：找命令（蓝色便利贴）**

每个事件都是被某个"命令"触发的，在事件左边贴上触发它的命令：

```
[提交订单] → 订单已创建
[锁定库存] → 库存已锁定
[发起支付] → 支付已完成
```

**Step 4：找聚合（黄色便利贴）**

命令作用在某个业务对象上，这个对象就是聚合的候选：

```
[提交订单] → (订单) → 订单已创建
[锁定库存] → (库存) → 库存已锁定
[发起支付] → (支付单) → 支付已完成
```

**Step 5：识别限界上下文**

观察聚合的聚集情况，把概念相近、语言一致、生命周期相关的聚合圈在一起，每个圈就是一个限界上下文候选：

```
┌─────────────────┐   ┌──────────────┐   ┌──────────────┐
│   订单上下文     │   │  库存上下文   │   │  支付上下文   │
│  订单 / 优惠券  │   │    库存      │   │    支付单    │
└─────────────────┘   └──────────────┘   └──────────────┘
```

**Step 6：标注协调关系**

圈出来之后，看各上下文之间如何交互，对应到 DDD 的集成模式：

```
订单上下文 → 库存上下文：订单创建后发命令锁库存（上下游关系）
订单上下文 → 支付上下文：订单确认后触发支付（上下游关系）
订单上下文 → 物流系统（第三方）：无条件服从 + ACL 转换
```

**Event Storming 是发现，划分领域是决策**，六步走完后直接过渡：

- **限界上下文候选 → 正式划定领域边界**：Step 5 圈出的上下文是草稿，这一步决策哪些合并（概念高度重叠）、哪些拆分（职责太杂），结果就是各个正式子域
- **聚合候选 → 确认核心域/支撑域/通用域**：看哪些聚合承载了最核心的业务价值。比如电商里"订单"是核心竞争力（核心域），"支付"接第三方（通用域），"优惠券"是定制逻辑（支撑域）
- **协调关系草图 → 正式确定集成模式**：Step 6 是初步判断，这一步正式决定每对上下文用哪种模式（Customer-Supplier / Conformist / ACL / Open API），并明确输入输出契约

两个阶段没有硬边界，通常是同一个工作坊里连续进行的。

### 1. 划分领域(Domain)

基于 Event Storming 的产出，对领域边界、优先级和集成关系做正式决策。

#### 识别领域的优先级和类别

将划分好的领域分为**核心域**，**支撑域**和**通用域**

- 核心域：这个需求中我们核心要实现的能力，最重要的部分
- 支撑域：起到支撑性作用的，需要结合团队技术栈定制的部分
- 通用域：业界都在用的，共通的东西，可以直接拿来用

#### 确定领域边界

##### 限界上下文

限界上下文定义的是**一套领域模型 + 通用语言（Ubiquitous Language）+ 功能集合**三者绑定的语义边界。三者相互绑定：同一个词（比如"用户"）在不同限界上下文里可以对应完全不同的模型和功能。限界上下文的核心价值是让这套语言和模型在边界内保持一致，不被外部污染。

##### 确定好输入和输出

结合需求描述，从 Use Case 角度描述哪些输入和输出是这个领域提供的

#### 确定领域集成关系

领域和领域之间如何协调和关联的：

- 上下游关系：下游可以向上游提需求，上游会考虑下游的诉求，双方有协商空间（Customer-Supplier）
- 无条件服从：上游根本不在乎下游（比如调用第三方平台），下游只能无条件跟着上游的模型走，没有任何谈判余地（Conformist）
- ACL转换：制造ACL层，领域都只感知自己的领域模型，制造转换层进行转化
- 提供API：领域提供稳定的标准API给调用方使用

### 2. 领域建模

基于 Event Storming 产出的聚合候选和通用语言草稿，在限界上下文内做精化建模。

#### 输出概念和规则列表

把新产生的概念和规则作为通用业务语言固化下来，输出到一个列表里，比如：

| 领域术语 | 什么意思                                                     |
| -------- | ------------------------------------------------------------ |
| Task     | 用户提出的需要Agent完成的任务                                |
| Turn     | 提问的轮次，在一轮中以用户提问开始，以Agent给出最终答案为结束，此为一个Turn |

| 规则         | 描述                                           |
| ------------ | ---------------------------------------------- |
| 支付金额规则 | 用户支付的金额必须等于商品金额加上物流费用金额 |

#### 抽聚合根

**围绕限界上下文**将领域术语变成一个个对象。对象分两类：
- **Entity（实体）**：有唯一标识，生命周期内状态可变（比如订单）
- **Value Object（值对象）**：无唯一标识，通过属性值判断相等，不可变（比如金额、地址）

有一些对象需要有共同的生命周期，状态和事务一致性边界，此时要抽**聚合根**给这些对象。围绕聚合根设计一个聚合，含有哪些成员（可能是Entity，可能是值对象，可能是其他聚合根的引用ID，**切忌不能直接引用其他聚合根对象**），成员方法的签名契约

#### 在领域内实体上应用规则

规则可能存在于聚合根的构建方法，存在于一个执行功能的成员方法，也可能是配置文件等，放在合理的位置，确保规则没有遗漏

#### 构建领域内聚合的协调关系

**围绕限界上下文**处理领域内聚合根的协调关系：

- 抽出一个**领域服务**来协调不同的聚合根，实现一些单聚合无法完成的能力和行为
- 通过**领域事件**协调：一个聚合生产事件，另一个消费事件。领域事件除了用于领域内聚合协调，还是**跨限界上下文解耦集成**的主要手段——上下文 A 发布事件，上下文 B 订阅，两者不直接依赖

#### 构建领域与外部的交互能力

这一步要建立领域与外部的互通，采用 Port 与适配器方法：

- 领域调用外部，只能依赖抽象接口，需要通过适配器去实现接口
- 外部调用领域能力，外部也是调用一个接口，然后需要为某个领域抽出一个 Application 层去实现这个接口，处理非领域的核心逻辑，比如DTO转领域对象、安全、应用事务处理等。同领域的 Application 调用领域模型对象
- **聚合的持久化**通过 **Repository（仓储）** 模式处理：领域层只依赖 Repository 接口，基础设施层负责实现（对接数据库），聚合本身不感知存储细节

### 3. 领域编排

结合整体需求目标考虑提供一个横向的编排应用层，编排和调用各个领域提供的 Port

### 4. 整体软件架构

DDD 落地通常对应四层，从上到下依赖单向流动：

```
Interface 层     （API / Controller，接收外部请求）
      ↓
Application 层   （用例编排，DTO 转换，事务边界）
      ↓
Domain 层        （聚合、领域服务、领域事件，纯业务逻辑）
      ↓
Infrastructure 层（Repository 实现、外部适配器、消息队列等）
```
## DDD一些错误的理解记录

### Event Storming
- 站在用户角度，而不是技术角度
- 列举的是影响业务事实，已发生的事件，而不是流程
- 运用事件思维，而不是流程思维。比如“TaskSubmitted”是事件，“Prompt已生成”是技术流程
- 列举的事件是因果关系的证据，流程是因果关系的解释
- 用事件反推系统，而不是用流程去定义系统
- 流程是“人脑解释世界的方式”，事件是“世界真实发生的记录”
- 准备做某事不是事件，已经发生并改变了世界才是事件
- 换实现，影响流程，但绝不应该影响事件
