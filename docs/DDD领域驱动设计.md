# DDD领域驱动设计笔记

## DDD整体目标

围绕以需求，团队协作为中心，从业务建模到软件架构落地的一套思考体系

## 实施过程

### 1. 划分领域(Domain)

首先思考需求的功能流程，以流程节点为参考，将待解决的问题划分为几个独立的子问题空间领域，每个领域都有自己的限界上下文，领域之间完全是独立的，是可独立迭代和发展的

#### 识别领域的优先级和类别

将划分好的领域分为**核心域**，**支撑领**和**通用域**

- 核心域：这个需求中我们核心要实现的能力，最重要的部分
- 支撑域：起到支撑性作用的，需要结合团队技术栈定制的部分
- 通用域：业界都在用的，共通的东西，可以直接拿来用

#### 确定领域边界

##### 限界上下文

哪些是这个领域需要实现的功能，哪些不是，必须明确

##### 确定好输入和输出

结合需求描述，从User Case角度描述哪些输入和输出是这个领域提供的

#### 确定领域协调关系

领域和领域之间如何协调和关联的：

- 上下游关系：上游改变，下游需要适配，上下游可以协调（TODO：有点概念化，能否再具体讲到执行层面）
- 无条件服从：上游变动，下游必须无条件跟着改（TODO：感觉这一条和上一条区别就是上面那个可以商量？）
- ACL转换：制造ACL层，领域都只感知自己的领域模型，制造转换层进行转化
- 提供API：；领域提供稳定的标准API给调用方使用

### 2. 领域建模

用业务语言，以功能目标为导向，描述这个领域内有哪些业务实体，需要满足什么规则和限制

#### 输出概念和规则列表

把新产生的概念和规则作为通用业务语言固化下来，输出到一个列表里，比如：

| 领域术语 | 什么意思                                                     |
| -------- | ------------------------------------------------------------ |
| Task     | 用户提出的需要Agent完成的任务                                |
| Turn     | 提问的轮次，在一轮中以用户提问开始，以Agent给出最终答案为结束，此为一个Turn |

| 规则         | 描述                                           |
| ------------ | ---------------------------------------------- |
| 支付金额规则 | 用户支付的金额必须等于商品金额加上物流费用金额 |

#### 抽聚合根

**围绕限界上下文**将领域术语变成一个个对象，有一些对象需要有共同的生命周期，状态和事务一致性边界，此时要抽**聚合根**给这些对象。围绕聚合根设计一个聚合，含有哪些成员（可能是Entity，可能是值对象，可能是其他聚合根的引用ID，**切忌不能直接引用其他聚合根对象**），成员方法的签名契约

#### 在领域内实体上应用规则

规则可能存在于聚合根的构建方法，存在于一个执行功能的成员方法，也可能是配置文件等，放在合理的位置，确保规则没有遗漏

#### 构建领域内聚合的协调关系

**围绕限界上下文**处理领域内聚合根的协调关系：

- 抽出一个领域服务来协调不同的聚合根，实现一些单聚合无法完成的能力和行为
- 通过领域事件的桥接对象，一个生产事件，一个消费事件

#### 构建领域与外部的交互能力

这一步要建立领域与外部的互通，采用PORT与适配器方法

- 领域调用外部，只能依赖抽象接口，需要通过适配器去实现接口
- 外部调用领域能力，外部也是调用一个接口，然后需要为某个领域抽出一个application层去实现这个接口，处理非领域的核心逻辑，比如DTO转领域对象，安全，应用事务处理等。同领域的Application调用领域模型对象

### 领域编排

结合整体需求目标考虑提供一个横向的编排应用层，编排和调用各个领域提供的PORT
